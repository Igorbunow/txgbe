# Архитектура проекта (ARCHITECTURE.md)

[ English ](ARCHITECTURE.md) | [ Русский ](ARCHITECTURE_ru.md)

## Обзор
Этот репозиторий содержит внедеревовой (out-of-tree) модуль ядра Linux `txgbe` для сетевых карт PCIe 10GbE на базе WangXun (например, LR-LINK LRES1002PF-2SFP+). Основная цель — **совместимость с несколькими LTS-ядрами** без регрессий.

Поддерживаемые версии ядер:
- 4.19, 5.4, 5.10, 5.15, 6.1, 6.6, 6.12, 6.18

## Структура репозитория
- `src/` — исходные коды драйвера и слой совместимости
  - `txgbe_main.c` — основная реализация netdev, PCI probe/remove, связка с трактом данных
  - `txgbe_ethtool.c` — операции ethtool (сигнатуры функций зависят от версии ядра)
  - `txgbe_lib.c` — распределение очередей/векторов, интеграция с NAPI (частые изменения API)
  - `txgbe_ptp.c` — поддержка PTP / аппаратных меток времени (API ядра сильно различается)
  - `kcompat*.h`, `kcompat*.c` — слой совместимости (флаги функций + бэкпорты)
  - `generated.sh`, `kcompat-generator.sh` — обнаружение функций и генерация определений
- `scripts/` — скрипты сборки / вспомогательные скрипты (если есть)
- `*.spec`, `kmod-*.spec` — артефакты сборки RPM (от производителя)

## Система сборки
Сборка выполняется вне дерева ядра, но интегрируется с kbuild:
- Типичная сборка:
  `make -C "$KERNELDIR" M="$PWD/src" modules`

Ключевое требование: `src/kcompat_generated_defs.h` должен быть сгенерирован для того же дерева/заголовков ядра, которые используются для компиляции.

## Система совместимости (kcompat)
### Почему она существует
API ядра для сетевых драйверов меняется между выпусками LTS (операции netdev, ethtool, помощники XDP, NAPI, PCI/MSI API и т. д.). Чтобы не загромождать код драйвера проверками версий, используется слой совместимости.

### Компоненты
1. **Обнаружение функций**
   - `src/generated.sh` выбирает подходящий заголовок конфигурации (`autoconf.h`) и пути включения.
   - Запускает `src/kcompat-generator.sh`, который сканирует заголовки ядра и выдает флаги функций.
   - Выходной файл: `src/kcompat_generated_defs.h`

2. **Цепочка включения определений**
   - `kcompat_generated_defs.h` (автогенерация) -> включается в `kcompat_defs.h`
   - `kcompat.h` предоставляет консолидированный интерфейс для исходных кодов драйвера.

3. **Бэкпорты и обертки**
   - `kcompat_impl.h` содержит макросы-обертки и реализации бэкпортов, защищенные флагами `HAVE_*` / `NEED_*`.

### Политика проекта
- Предпочитайте проверки `HAVE_*` / `NEED_*` логике на основе `LINUX_VERSION_CODE`.
- Если проверки версии ядра неизбежны из-за бэкпортов в дистрибутивах, изолируйте их в `kcompat_impl.h` с четким обоснованием.

## Как добавить поддержку новой версии ядра
1. Соберите модуль для этого дерева/заголовков ядра и найдите первую ошибку.
2. Убедитесь, что `kcompat_generated_defs.h` перегенерирован для этого ядра (чистая сборка).
3. Выберите минимальное исправление:
   - Отсутствует флаг функции -> обновите правило генератора.
   - Несоответствие сигнатуры -> добавьте/настройте обертку в `kcompat_impl.h`.
   - Только если неизбежно -> защитите код в исходниках драйвера флагами HAVE_/NEED_.
4. Пересоберите всю матрицу ядер (все поддерживаемые версии).
5. Задокументируйте новые флаги или обертки.

## Предотвращение регрессий (обязательно)
- Никогда не проверяйте только на одном ядре.
- Изменения должны быть минимальными и локальными.
- Не добавляйте «новые функции» для поддержки новых ядер, если они не требуются для сборки.
- Не добавляйте `kcompat_generated_defs.h` в git; добавьте его в `.gitignore`.

## Рекомендуемая CI (GitHub Actions)
Реализуйте матрицу сборки, которая компилирует `src/` для набора деревьев заголовков ядра (или образов контейнеров), соответствующих поддерживаемым версиям. CI должна завершаться ошибкой при любой ошибке сборки.

## Заметки по повышению производительности на LoongArch / Loongson

Проверка на Loongson-3A6000 выявила несколько критичных для производительности областей, которые могут быть полезны и на других архитектурах, но должны оставаться опциональными, если не доказана их необходимость для платформы.

### IRQ, RSS и привязка к процессору

Драйвер распределяет MSI-X q_vectors для каждого адаптера. На двухпортовой сетевой карте 10G стандартное сопоставление q_vector и CPU может поместить оба порта на один и тот же набор ядер. Это может ограничить общую пропускную способность, даже если каждый порт может достигать 10G независимо.

Для 8-ядерной системы Loongson-3A6000 проверенная конфигурация отладки использует:

```text
RSS=4,4
txgbe_force_irq_affinity=1
txgbe_port_affinity_spread=1
```

Ожидаемое сопоставление:
```text
порт 0 q_vectors -> CPU0..CPU3
порт 1 q_vectors -> CPU4..CPU7
```
Это предотвращает конкуренцию обоих портов за одни и те же ядра softirq/IRQ.

### Порог обратной записи Tx (Write-back threshold)

Оригинальный код устанавливал высокий порог обратной записи дескрипторов Tx даже для режимов с низким или отключенным ограничением прерываний. На LoongArch это может сделать поведение завершения Tx более нестабильным при высокой нагрузке. Параметр `txgbe_tx_wthresh_safe` сохраняет старое поведение по умолчанию на платформах, отличных от LoongArch, позволяя при запуске на LoongArch использовать более безопасный порог.

Политика:
- Не делайте настройки для Loongson глобальными без тестирования на других архитектурах.
- Оставляйте параметры времени выполнения доступными для A/B тестирования.
- Держите диагностическое логирование отключенным по умолчанию.

### Опрос статуса SFP

Некоторые пути опроса медных SFP / внутренних PHY могут вызывать падения линка или ложные переходы состояния на чувствительных системах. Параметр `txgbe_sfp_status_poll` позволяет отключить этот опрос во время диагностики. Это не универсальное решение: отключение опроса может снизить скорость реакции на горячую замену или изменение статуса.

Используйте `txgbe_link_diag=1` или `2` только при сборе доказательств ошибок состояния линка.

### Требования к методологии тестирования

Тесты пропускной способности на двух портах должны быть детерминированными:
- привязывайте каждого клиента с помощью `iperf3 -B <исходный-ip>`;
- используйте разные IP-адреса назначения/подсети для каждого порта;
- проверяйте `ip route get <dst> from <src>` перед тестированием;
- проверяйте `/proc/interrupts` и `/proc/irq/*/smp_affinity_list`;
- следите за счетчиками `ethtool -S`, особенно `rx_crc_errors`, `rx_no_buffer_count`, `rx_missed_errors`, `tx_timeout_count` и счетчиками управления потоком.

Быстро растущие `rx_crc_errors` следует в первую очередь рассматривать как проблему физического линка: поменяйте местами модули SFP, кабели и порты удаленного коммутатора/сервера перед изменением логики драйвера.
