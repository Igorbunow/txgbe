## Почему существует этот форк

Пакет драйверов от производителя был опубликован под лицензией GPL, однако на практике поддержка различных веток ядер часто ломалась при быстром добавлении новых версий ядра.

Этот форк сосредоточен на:
- Сохранении **работоспособности существующих ядер** при добавлении поддержки новых.
- Внесении **минимальных изменений, ориентированных на совместимость** (никаких «попутных рефакторингов»).
- Использовании существующего подхода **kcompat / feature-detection** (обнаружение доступных функций), чтобы избежать «угадывания» API исключительно на основе версии ядра.

### Последнее обновление совместимости (устаревание hwmon начиная с 5.15+)

Драйвер больше не полагается исключительно на устаревшую функцию `hwmon_device_register()`, если доступны более новые API.

Текущий процесс регистрации:
- сначала пробуется `hwmon_device_register_with_info()`
- при необходимости используется резервный вариант `hwmon_device_register_with_groups()`
- возврат к устаревшему `hwmon_device_register()` происходит только в случае неудачи первых двух вариантов

Детали реализации:
- используется обнаружение функций kcompat (`HAVE_HWMON_DEVICE_REGISTER_WITH_INFO`, `HAVE_HWMON_DEVICE_REGISTER_WITH_GROUPS`)
- добавлена безопасная очистка для путей с ошибками hwmon (обработка `IS_ERR_OR_NULL`)
- сохранены существующие атрибуты температуры sysfs драйвера (`temp0_input`, `temp0_alarmthresh`, `temp0_dalarmthresh`)

Статус проверки:
- проверено во время выполнения с помощью `insmod` на 5.15.148-tegra (Jetson AGX Orin): загрузка/выгрузка модуля работает, поднятие/опускание линка работает, изменение MTU на 9000 работает
- проверено матричной сборкой:
  3.18.140, 4.1.52, 4.4.302, 4.9.337, 4.14.336, 4.19.325, 5.4.302, 5.10.250, 5.15.200, 6.1.163, 6.6.124, 6.12.71, 6.18.10

## Скрипт подготовки заголовочных файлов ядра (для нескольких веток)

Скрипт `tools/prepare_kernels_arm64.sh` скачивает архивы исходных кодов ядер и подготавливает заголовочные файлы (с помощью `modules_prepare`) в директорию `/opt/kernels/<версия>/build-<ARCH>` по умолчанию. Он также поддерживает выбор произвольных LTS-веток, использование альтернативных источников ядер и сохранение директорий сборки, специфичных для конкретного компилятора.

Ключевые особенности:
- Определяет последние версии патчей для каждой ветки (из `releases.json`, с откатом на CDN при его отсутствии).
- Поддерживает альтернативный базовый URL для загрузки через `KERNEL_SOURCE_BASE` / `--kernel-source-base`.
- Записывает метаданные компилятора в каждой директории сборки (`.toolchain-info`) для выявления несовпадений при последующих запусках.

### Примеры использования

Подготовка стандартных поддерживаемых веток:

```bash
sudo tools/prepare_kernels_arm64.sh

```

Подготовка только выбранных веток:

```bash
sudo tools/prepare_kernels_arm64.sh --kernels-list "4.14 4.19 5.10"

```

Использование последнего доступного уровня патчей для каждой выбранной ветки:

```bash
sudo tools/prepare_kernels_arm64.sh --latest-all --kernels-list "4.4 4.9 4.14"

```

Использование локального lock-файла релизов:

```bash
sudo RELEASES_JSON=/etc/kernel-build/releases.json tools/prepare_kernels_arm64.sh

```

Тестовый запуск (показать только план действий):

```bash
sudo tools/prepare_kernels_arm64.sh --dry-run --kernels-list "4.14 4.19"

```

Вывод справочной таблицы upstream LTS-релизов:

```bash
tools/prepare_kernels_arm64.sh --lts-reference

```

### Политика пересборки с учетом компилятора

Если директория сборки уже существует, скрипт проверяет, каким компилятором она была подготовлена, и сравнивает с выбранным в данный момент компилятором. Сравнение «мягкое» для мажорной версии и строгое для полной версии. При обнаружении несовпадения можно выбрать дальнейшие действия:

* Prompt (по умолчанию): интерактивный выбор — пересобрать или оставить как есть.
* `--mismatch-rebuild`: удалить директорию сборки и пересобрать в неинтерактивном режиме.
* `--mismatch-skip`: сохранить директорию сборки и пропустить шаг в неинтерактивном режиме.
* `--quiet`: отключить запросы; требует указания одной из неинтерактивных политик выше.

Примеры:

```bash
sudo tools/prepare_kernels_arm64.sh --mismatch-rebuild
sudo tools/prepare_kernels_arm64.sh --mismatch-skip
sudo tools/prepare_kernels_arm64.sh --quiet --mismatch-rebuild

```

### Директории сборки, специфичные для компилятора

Если вам нужны параллельные деревья для разных наборов инструментов (toolchains), включите создание поддиректории для конкретного компилятора:

```bash
sudo tools/prepare_kernels_arm64.sh --toolchain-subdir

```

Это изменит структуру директорий сборки на:

```
/opt/kernels/<версия>/build-<ARCH>/<метка-компилятора>/

```

### Альтернативный источник ядер

Использование другого CDN/корневого пути для архивов:

```bash
sudo tools/prepare_kernels_arm64.sh --kernel-source-base [https://cdn.kernel.org/pub/linux/kernel](https://cdn.kernel.org/pub/linux/kernel)

```

## Запуск 10G на двух портах Loongson-3A6000

В этом разделе задокументирована проверенная конфигурация для **Loongson-3A6000 / LoongArch64** с сетевой картой LR-LINK LRES1002PF-2SFP+, где оба SFP+ порта работают на полной скорости канала (line rate).

Протестированное окружение:

* Процессор: Loongson-3A6000 / LA664, 8 доступных ядер
* Ядро: ALT Linux LoongArch64 6.18.x
* Компилятор: GCC 14.x для loongarch64
* PCIe: 8.0 GT/s x8, доступная пропускная способность около 63 Гбит/с
* Сетевая карта: LR-LINK LRES1002PF-2SFP+ / WangXun txgbe, два порта SFP+
* Наблюдаемый результат: тест `iperf3` с привязкой к двум портам достигает около **9.90 Гбит/с на порт**

### Важные выводы по результатам запуска

Карта может работать с полной пропускной способностью обоих портов на Loongson, однако следующие проблемы могут создать иллюзию узкого места в драйвере или PCIe:

1. **Ошибки маршрутизации/привязки при тестировании.** Каждый клиент `iperf3` должен быть привязан через флаг `-B` и использовать соответствующий адрес узла (peer). Не тестируйте оба порта на один и тот же IP-адрес назначения.
2. **Вмешательство NetworkManager.** Ручная конфигурация через `ip addr` может сбрасываться после падения линка (link flap) или пересоздания сетевого устройства. Используйте неуправляемые (unmanaged) устройства для детерминированных тестов или создайте постоянные профили NetworkManager.
3. **Перекрытие прерываний (IRQ) / RSS на процессоре.** С двумя 10G портами на 8-ядерной системе используйте 4 очереди на порт и распределите порты между CPU0-3 и CPU4-7.
4. **Проблемы физического линка.** Быстрый рост `rx_crc_errors` указывает на проблемы с SFP/кабелем/удаленным портом, а не с RSS или PCIe. Поменяйте SFP/кабели/удаленные порты местами перед внесением изменений в драйвер.
5. **Не используйте `InterruptThrottleRate=0` в качестве первого шага при тюнинге.** На Loongson адаптивный режим с безопасным порогом обратной записи Tx (write-back threshold) оказался более стабильным при тестировании.

### Загрузка модуля для отладки / запуска

Для запуска на Loongson сначала используйте `insmod`, чтобы каждый параметр был явно указан и его было легко изменять между тестами:

```bash
sudo systemctl stop irqbalance 2>/dev/null || true
sudo nmcli dev set enp8s0f0 managed no 2>/dev/null || true
sudo nmcli dev set enp8s0f1 managed no 2>/dev/null || true

sudo rmmod txgbe 2>/dev/null || true
sudo insmod src/txgbe.ko \
  RSS=4,4 \
  InterruptThrottleRate=1,1 \
  AtrSampleRate=20,20 \
  irq_nobalance=0 \
  txgbe_perf_diag=2 \
  txgbe_force_irq_affinity=1 \
  txgbe_port_affinity_spread=1 \
  txgbe_tx_wthresh_safe=1 \
  sfp_i2c_fix=1 \
  txgbe_sfp_status_poll=0 \
  txgbe_link_diag=1

```

Ожидаемое распределение прерываний для `RSS=4,4` на 8-ядерном хосте Loongson:

```text
enp8s0f0-TxRx-0..3 -> CPU0..CPU3
enp8s0f1-TxRx-0..3 -> CPU4..CPU7

```

Проверьте это с помощью:

```bash
cat /proc/interrupts | grep -E 'CPU|enp8s0f0|enp8s0f1'
for irq in $(grep -E 'enp8s0f0|enp8s0f1' /proc/interrupts | awk -F: '{gsub(/ /,"",$1); print $1}'); do
    echo "$irq $(cat /proc/irq/$irq/smp_affinity_list)"
done

```

`txgbe_sfp_status_poll=0` — это диагностический/стабилизирующий обходной путь для систем, где опрос внутреннего PHY SFP вызывает падения линка. Это может снизить отзывчивость при горячем подключении (hotplug) и опросе статуса. Если линк стабилен при включенном опросе, предпочтительнее использовать устаревшее значение по умолчанию `txgbe_sfp_status_poll=1`.

### Детерминированный тест iperf3 для двух портов

Пример для сервера с двумя сетевыми картами, названными `eth2` и `eth6`:

```bash
sudo IF0=eth2 IF1=eth6 \
  IP0=10.0.20.30/24 IP1=10.0.21.31/24 \
  BIND0=10.0.20.30 BIND1=10.0.21.31 \
  tools/txgbe_server_dual_iperf_bound.sh

```

Пример для клиента с портами txgbe на Loongson, названными `enp8s0f0` и `enp8s0f1`:

```bash
sudo IF0=enp8s0f0 IF1=enp8s0f1 \
  IP0=10.0.20.20/24 IP1=10.0.21.20/24 \
  SRC0=10.0.20.20 SRC1=10.0.21.20 \
  DST0=10.0.20.30 DST1=10.0.21.31 \
  TIME=300 PARALLEL=4 \
  tools/txgbe_client_dual_iperf_bound.sh

```

Прежде чем доверять показателям пропускной способности, убедитесь, что маршрутизация разделена правильно:

```bash
ip route get 10.0.20.30 from 10.0.20.20
ip route get 10.0.21.31 from 10.0.21.20

```

Ожидаемый вывод должен содержать разные устройства, например:

```text
10.0.20.30 from 10.0.20.20 dev enp8s0f0 ...
10.0.21.31 from 10.0.21.20 dev enp8s0f1 ...

```

### Тестовый профиль неуправляемого интерфейса (unmanaged) в NetworkManager

Для временных лабораторных тестов скопируйте пример конфигурации и перезагрузите NetworkManager:

sudo cp docs/examples/99-txgbe-unmanaged.conf /etc/NetworkManager/conf.d/99-txgbe-unmanaged.conf

### Релизная установка с параметрами modprobe

После проверки параметров через `insmod`, установите модуль в дерево модулей запущенного ядра и используйте директорию `/etc/modprobe.d/` для сохранения постоянных параметров:

```bash
sudo install -D -m 0644 src/txgbe.ko /lib/modules/$(uname -r)/extra/txgbe/txgbe.ko
sudo depmod -a

```

Создайте `/etc/modprobe.d/txgbe-loongson.conf`:

```conf
# LR-LINK LRES1002PF-2SFP+ on Loongson-3A6000.
# Keep diagnostic verbosity disabled for normal operation.
options txgbe RSS=4,4 InterruptThrottleRate=1,1 AtrSampleRate=20,20 irq_nobalance=0   txgbe_force_irq_affinity=1 txgbe_port_affinity_spread=1 txgbe_tx_wthresh_safe=1   sfp_i2c_fix=1 txgbe_sfp_status_poll=0 txgbe_perf_diag=0 txgbe_link_diag=0

```

Затем загрузите стандартным способом:

```bash
sudo modprobe -r txgbe 2>/dev/null || true
sudo modprobe txgbe
cat /sys/module/txgbe/parameters/RSS 2>/dev/null || true
dmesg -T | grep -i txgbe | tail -n 100

```

Если опрос статуса SFP стабилен на вашем оборудовании, измените `txgbe_sfp_status_poll=0` на `txgbe_sfp_status_poll=1` для лучшего поведения статуса линка в legacy-режиме.

## Слой совместимости (kcompat) — важно

В этом драйвере используется слой совместимости, предназначенный для обнаружения функций ядра и выбора правильных обёрток (wrappers) / бэкпортов для каждого конкретного ядра.

Общие правила:

* Автогенерируемые дефайны (defines) совместимости создаются во время сборки для **точных** заголовочных файлов целевого ядра.
* **Не** коммитьте автосгенерированные заголовки в git.
* Предпочитайте флаги функций (`HAVE_`/`NEED_`) угадываниям на основе версий.

Смотрите `AGENTS.md` и `ARCHITECTURE.md` в этом репозитории для ознакомления с правилами и структурой проекта.

## Устранение неполадок

Этот драйвер опирается на автосгенерированный заголовочный файл совместимости (`kcompat_generated_defs.h`), созданный для **точных** заголовочных файлов целевого ядра, указанных в `KERNELDIR`. Многие «загадочные» ошибки сборки вызваны несовпадением между ядром, для которого вы компилируете, и ядром, использованным для генерации дефайнов kcompat.

### 1) Симптом: ошибки сборки после смены `KERNELDIR`

**Типичные ошибки**

* `error: too few arguments to function ‘netif_napi_add’`
* `error: ‘struct ethtool_rxfh_param’ has no member named ...`
* `error: implicit declaration of function ...`
* `warning treated as error: incompatible pointer type ...`

**Наиболее вероятная причина**
`kcompat_generated_defs.h` был сгенерирован для другого дерева ядра, отличного от текущего `KERNELDIR`.

**Решение**
Выполните чистую пересборку для нового `KERNELDIR`:

```bash
make -C "$KERNELDIR" M="$PWD/src" clean
make -C "$KERNELDIR" M="$PWD/src" modules -j"$(nproc)"

```

Если система сборки всё ещё использует устаревшие файлы, принудительно запустите полную пересборку:

```bash
make -B -C "$KERNELDIR" M="$PWD/src" modules -j"$(nproc)"

```

---

### 2) Симптом: `kcompat_generated_defs.h` «застрял» и не перегенерируется

**Типичное поведение**

* `make kcompat-generated` (или шаг генерации) не выполняется даже после изменения `KERNELDIR`
* Вы видите одни и те же ошибки на разных ядрах

**Решение**
Убедитесь, что сгенерированный заголовочный файл удалён и пересобран:

```bash
rm -f src/kcompat_generated_defs.h
make -C "$KERNELDIR" M="$PWD/src" modules -j"$(nproc)"

```

---

### 3) Симптом: генератор не может найти заголовочный файл конфигурации ядра (`autoconf.h`)

**Типичные ошибки**

* `fatal error: generated/autoconf.h: No such file or directory`
* `fatal error: linux/autoconf.h: No such file or directory`

**Причина**
`KERNELDIR` не указывает на подготовленное дерево сборки/заголовочных файлов ядра, или целевое ядро не было сконфигурировано.

**Решение**
Укажите в `KERNELDIR` правильную директорию сборки (заголовочные + сгенерированные файлы).
Для полных деревьев ядра выполните как минимум:

```bash
make olddefconfig
make prepare modules_prepare

```

(Точные команды могут отличаться в зависимости от того, как ваш дистрибутив упаковывает заголовочные файлы ядра.)

---

### 4) Симптом: компиляция работает для одного ядра, но завершается ошибками API/сигнатур на другом

**Причина**
Драйвер использует обнаружение функций ядра. Если сборка не была полностью очищена или используются неправильные заголовочные файлы, флаги `kcompat` могут не совпадать с фактическим API целевого ядра.

**Решение**

* Выполняйте чистую пересборку для каждого ядра (см. пункт #1).
* Убедитесь, что вы не смешиваете заголовочные файлы одного ядра с директорией сборки другого.

---

### 5) Как проверить, для какого ядра был сгенерирован `kcompat`

Если ваш `kcompat_generated_defs.h` содержит строку происхождения (что рекомендуется), проверьте её:

```bash
grep -nE "KSRC|KERNEL|Autogenerated|generated" src/kcompat_generated_defs.h | head -n 20

```

Если такой строки нет, самый безопасный подход остаётся прежним: удалите сгенерированный заголовочный файл и пересоберите (см. пункт #2).

---

### 6) Симптом: кросс-компиляция завершается ошибкой из-за отсутствующего toolchain или неверного `ARCH`

**Типичные ошибки**

* `aarch64-linux-gnu-gcc: command not found`
* `unknown register name ...` / `wrong ELF class ...`

**Решение**

* Убедитесь, что ваш кросс-компилятор установлен, и префикс `CROSS_COMPILE` указан верно.
* Убедитесь, что `ARCH` соответствует целевой архитектуре.

Пример (arm64):

```bash
export ARCH=arm64
export CROSS_COMPILE=aarch64-linux-gnu-
export KERNELDIR=/path/to/kernel/tree
make -C "$KERNELDIR" M="$PWD/src" modules -j"$(nproc)"

```

Пример (x86):

```bash
export ARCH=x86
export CROSS_COMPILE=""
export KERNELDIR=/path/to/kernel/tree
make -C "$KERNELDIR" M="$PWD/src" modules -j"$(nproc)"

```

---

### 8) Симптом: iperf3 "Connection refused" или "No route to host" во время тестирования

**Причина**
Системный брандмауэр (firewall) блокирует тестовый трафик.

**Решение**
Настройте брандмауэр для разрешения трафика на тестовых интерфейсах. Подробные инструкции см. в [FIREWALL_ru.md](docs/FIREWALL_ru.md).
